# Size Score Workflow

This workflow uses an ensemble of LLM models to evaluate English lemmas
for inclusion and sizing in a spell-checker wordlist.  The central
question for each lemma is: should it be in the wordlist, and if so,
at what size tier?

## Size Categories

- **60** — Mainstream words any educated user expects a spell checker to
  accept (e.g. "emoji", "YouTube", "blockchain").
- **70** — Valid but specialized or less routine; dictionary-attested but
  not everyday (e.g. "tokenization", "proteobacterium").
- **80** — Passes inclusion but cannot be justified at 60 or 70.
- **99 (Excluded)** — Not suitable for the wordlist at all (misspellings,
  transient slang, extremely obscure terms).

Borderline flags (`60/70`, `70/80`, `incl/excl`) capture cases where the
model is uncertain between adjacent tiers.

## Quick Start

```
cd size_score
./init.sh              # one-time setup: creates data.db
./req <model>          # run evaluations (e.g. ./req gpt-5.2)
```

## Pipeline

```
init.sh          Set up data.db: schema, input data, views.
     ↓
./req <model>    Send batches of lemmas to LLMs via OpenRouter,
                 parse markdown-table responses, store results.
     ↓
populate_size_scores.py
                 Aggregate per-model votes into model_size_scores,
                 then run the ensemble to populate final_size_scores.
     ↓
candidates view  Identify lemmas that still need more model evaluations.
     ↓
(repeat)         req loops: populate scores → send next round →
                 until every lemma has enough confident coverage.
```

The `req` wrapper calls `python3 -m req` with the `req/` package in the
parent directory.  All paths (db, key, system prompt, config) resolve
relative to this working directory.

## req-config.py

Configures the generic `req` package for size evaluation:

- **`x_title`** — `'Corpus Size Scoring'` (required HTTP header)
- **`post_run`** — `['python3', 'populate_size_scores.py']` (runs between
  rounds to refresh scores and the candidates view)
- **`validate_row(row, input_row)`** — domain-specific validation:
  - Lemma matching: Unicode-normalized comparison ensures the LLM returned
    at least one matching lemma from the input.
  - Size normalization: converts "excluded"/"exclude" text to integer 99
    (basic int conversion is handled automatically by the core).
  - Borderline normalization: maps raw strings to canonical values
    (`''`, `60/70`, `70/80`, `80/99`).

CHECK constraints in `schema-local.sql` (`size in (60, 70, 80, 99)`,
`borderline in ('', '60/70', '70/80', '80/99')`) serve as a final safety
net — rows that fail INSERT are retried automatically.

## Input

Lemmas live in the `input` table (loaded from `test-input.sql`).  Each
has a uid, one or more comma-separated lemma forms, and a part of speech.
Examples span common words, brand names, abbreviations, scientific terms,
slang, chemical elements, and currency codes.

## System Prompt

Generated by `system_prompt.sh` from shared prompt fragments in
`../prompts/eval/`:

- **inc.md** — Inclusion/exclusion criteria: what makes a word suitable
  for the wordlist, and what disqualifies it.
- **size.md** — Size-tier definitions and the evaluation procedure
  (evaluate 60 first, then 70, then 80), plus borderline handling
  and typo-masking considerations.

The generated prompt tells the LLM to return a markdown table with
columns: `uid`, `lemmas`, `pos`, `size`, `borderline`, `size_notes`.

## Score Aggregation (populate_size_scores.py)

Reads `results_w_model` (results joined with run metadata) and, for
each (uid, model) pair:

- Converts borderline votes: 2/3 weight to the winning side, 1/3 to
  the losing side.
- Picks the model's chosen size via an upper-median of the weighted
  distribution.
- Writes per-model mass allocations (`s_60`, `s_70`, `s_80`, `s_99`)
  and upper/lower bounds to `model_size_scores`.
- Calls `determine_final_size()` from `size_decider.py` to compute the
  ensemble decision and writes it to `final_size_scores`.

## Ensemble Voting (size_decider.py)

A three-stage decision process using weighted votes from all models:

1. **Exclusion (size 99).**
   If GPT-5.2 puts ≥ 2/3 of its mass on 99 the lemma is excluded.
   DeepSeek can also exclude if ≥ 4/5 mass on 99 *and* corroborated by
   at least one other model.

2. **Size 60 decision.**
   GPT-5.2 has veto power: if it assigns zero mass to 60 the lemma
   cannot be 60.  Otherwise a weighted score
   `Σ weight × (mass_60 − mass_70+)` is computed; if it meets the 60%
   threshold the lemma is size 60.  Llama can provide a boost if the
   score is below threshold but Llama strongly favors 60.

3. **Size 70 vs 80.**
   A similar weighted score `Σ weight × (mass_60 + mass_70)` decides
   between 70 and 80 at a 60% threshold.  DeepSeek's size-80 mass gets
   a 30% bonus toward the 70 score (DeepSeek's 80 often acts like a
   conservative 70).

Model weights differ by stage and are defined in `size_decider.py`.
All arithmetic uses exact rational fractions (`fracs.py`) to avoid
floating-point rounding issues.

## Database Schema (task-specific tables)

Defined in `schema-local.sql`:

| Table | Purpose |
|---|---|
| `input` | Lemmas to evaluate (uid, lemmas, pos) |
| `results` | Parsed model judgments (uid, size, borderline, size_notes) with CHECK constraints |

Defined in `candidates.sql`:

| View | Purpose |
|---|---|
| `candidates` | Identifies lemmas needing more evaluations per model |

Computed by `populate_size_scores.py`:

| Table | Purpose |
|---|---|
| `model_size_scores` | Aggregated per-model mass distributions (s_60–s_99, lower, higher) |
| `final_size_scores` | Ensemble decision for each lemma (size, reason, scores) |

Common tables (`runs`, `requests`, `raw_data`, `errors`, `models`),
`results_w_model` view, and cost-tracking views are defined in the
`req/` package (`schema.sql` and `post.sql`).
